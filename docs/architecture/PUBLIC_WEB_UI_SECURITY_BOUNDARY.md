# PUBLIC_WEB_UI_SECURITY_BOUNDARY

Last updated: 2026-03-18

## 为什么“非常用端口”不算安全方案

- 端口变化只是在降低被随手猜到的概率，不提供认证、授权或加密。
- 当前 Web UI 已经可以触发提交、状态读取、review continue 和结果读取；一旦公网可达，单靠端口并不能阻止访问。
- 它不能替代：
  - HTTPS
  - 身份校验
  - 暴露面收口
  - 基础访问日志

## 为什么只暴露 Web UI

- 当前仓库的远程操作入口应尽量收敛成一个面，减少公网暴露点。
- Web UI 已经是现有人工操作的主壳；下一阶段只需要把它变成可远程自用，而不是再把 Job API 变成第二个公网入口。
- 只暴露 Web UI 更符合“单机自用工作台”的边界。

## Job API 为什么必须保持本机 / 内网

- 当前 Job API 没有面向公网的独立认证边界。
- 它直接承载：
  - `POST /jobs`
  - `GET /jobs/{job_id}/logs`
  - `POST /jobs/{job_id}/continue`
  - `GET /jobs/{job_id}/result-summary`
  - `GET /jobs/{job_id}/artifacts`
- 它还直接连接本地任务存储、事件日志、项目目录和 review 语义。
- 对这个仓库来说，Job API 更适合作为本机集成层，而不是公网产品接口。

## 最小认证边界建议

- 最小认证应放在公网入口层，不先做应用内账户系统。
- P2 当前已落地的最小方案是：
  - `Caddy`
  - `HTTPS`
  - `basic_auth`
  - 单用户或单组共享凭证
- 目标只是防止“任何人只要拿到 URL 就能直接进入 Web UI”。

## 凭证放置原则

- 不把凭证写进仓库。
- 不把凭证硬编码到 `web_ui.py`、前端页面或文档示例里。
- 凭证应放在仓库外的部署侧配置中，或放在 Windows 主机的安全凭证存储中。
- 仓库内允许存在的最多只是“如何读取部署侧凭证”的约定，不应存放真实值。
- P2 当前约定的最小凭证入口是环境变量：
  - `AUTODUB_PUBLIC_ENTRY_USERNAME`
  - `AUTODUB_PUBLIC_ENTRY_PASSWORD_HASH`
- 密码明文不入仓库；仓库内只保存哈希值的环境变量名。
- `public-entry` 的启动 / check-only 应在这两个环境变量缺失时直接失败，而不是以不完整凭证继续尝试启动。

## HTTPS / 反向代理 / 隧道层的角色边界

### HTTPS

- 负责链路加密。
- 不让 Web UI 以明文公网暴露。

### 反向代理 / 隧道层

- 负责公网暴露面
- 负责最小认证
- 负责把外部请求按路径分流到 Gateway (8880) 或 Next.js (3000)
- 可承担最基本的访问日志
- P2 当前由 `Caddy` 承担这一层，并关闭其 admin API。

### Web UI 应用本体

- 继续专注任务提交、审校和结果展示。
- 不把它当成第一道公网安全边界。

## 明确不做什么

- 不把安全方案简化成“换端口”
- 不做完整账户系统
- 不做组织权限 / 多用户 / RBAC
- 不把 Job API 直接暴露到公网
- 不做企业级零信任或完整 WAF 方案
- 不开始商业化安全设计
