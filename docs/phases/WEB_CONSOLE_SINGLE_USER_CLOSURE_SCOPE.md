# WEB_CONSOLE_SINGLE_USER_CLOSURE_SCOPE.md

## 1. 阶段唯一目标

本阶段只有一个目标：

将当前 AIVideoTrans Web Console 收口为“单用户可连续使用的完整闭环”，在现有 3 个前端页面基础上完成运行、review、使用说明和状态表达的最后对齐。

这不是第二批页面开发，也不是产品化扩张阶段。

## 2. 当前为什么需要这个阶段

首批 3 页已经达到第一批可用基线，但距离“单用户连续使用时不需要频繁 SSH 排障”的状态还差最后一层收口：

- 运行配置缺失时，前端表达还不够前置和清晰
- review 入口虽然可用，但需要正式定义为当前阶段的稳定方案
- 前端页面状态与现网运行态之间仍有少量错配和排障成本
- 用户还不够容易区分“配置问题”和“页面问题”

因此，下一阶段不应扩张页面数量，而应把单用户完整闭环收紧。

## 3. 本阶段必须做

- 运行配置收口
  - 明确单用户运行所需的 provider / API key / cookies / 前置条件
  - 缺失配置时，通过环境变量或本地配置提供，并在前端给出清晰、可执行的提示
  - 必要时加入更明确的前置检查或失败提示

- 单用户主链路收口
  - 只围绕现有 3 页：新建翻译、当前任务、项目详情
  - 把“新建 -> 运行 -> review -> 回看状态 -> 项目详情 -> 下载”收口为连续可用链路
  - 减少单用户正常使用时对 SSH 排障的依赖

- review 入口收口
  - 明确新前端与旧 Web UI 的职责边界
  - 明确“从新前端跳到旧 Web UI 处理 review”是当前阶段的正式方案
  - 明确哪些 review 能力是稳定承诺，哪些只是条件出现

- 使用与部署说明收口
  - 明确如何启动、如何检查配置是否齐全
  - 明确如何判断是现网配置问题还是页面问题
  - 保留必要部署注意事项，避免旧链路被误删

## 4. 本阶段明确不做

- 不做第二批页面
- 不做“我的项目”完整实现
- 不做设置中心化
- 不做登录/注册
- 不做数据库
- 不做多用户
- 不做商业化页面
- 不做 failed resume 系统
- 不做完整 review 中心重建
- 不做新的大架构升级

## 5. 当前重点收口问题

- 必要运行配置是否齐全，以及缺失时如何被前端正确表达
- review 入口链路是否作为当前阶段正式方案被清晰定义
- 单用户连续使用时，前端状态、日志、失败提示是否足够支撑自助判断
- 现网配置问题与页面问题的边界是否足够清楚

## 6. 验收标准

- 单用户在必要环境变量或本地配置齐全的前提下，可完成现有 3 页完整闭环，不需要频繁 SSH 排障
- 缺失关键运行配置时，前端能给出明确、可执行的提示，而不是只暴露模糊失败
- review 阶段入口、跳转、回看状态变化都可稳定复现
- 项目详情、结果摘要、白名单下载在主链路中保持可访问
- 使用与部署说明能帮助用户区分“配置问题”与“页面问题”
