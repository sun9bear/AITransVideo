# WEB_CONSOLE_SINGLE_USER_PRODUCTIZATION_SCOPE.md

## 1. 阶段唯一目标

本阶段只有一个目标：

将当前已经可用的单用户 Web Console 继续整理为“更像完整产品、但仍然是单用户基线”的稳定形态。

这一阶段只做单用户产品化整理，不进入多用户底座、登录体系、数据库或商业化扩张。

## 2. 为什么现在需要这个阶段

- 当前主流程已经跑通，三类 review 也都已经以前端为主入口
- 第二批轻量页面和第一轮 UX 优化已经完成，继续堆新功能的收益不如先做整理
- 现在最影响连续使用体验的，已经不是“有没有页面”，而是“文案是否一致、信息是否顺、入口是否清楚、结果是否好理解”
- 因此下一阶段更适合做单用户产品化收口，而不是提前跳到多用户或商业化方向

## 3. 本阶段必须做

- 统一用户可见文案与命名
  - 页面标题
  - 状态 / 阶段 / 审核文案
  - 成功 / 失败 / 空态文案
  - 结果下载命名
  - 用户可见按钮文字
- 统一核心页面信息架构
  - 新建翻译
  - 当前任务
  - 我的项目
  - 项目详情
  - 三类 review 页面
  - 设置页
- 整理结果区与下载区
  - 让结果命名更像用户结果
  - 让结果摘要更清楚
  - 让结果 / 下载 / 日志关系更顺
- 整理入口分层
  - 主入口
  - fallback 入口
  - internal / 弱入口
- 整理设置页说明
  - 保持说明页定位不变
  - 更清楚表达当前版本、支持项、不支持项、默认项和系统级配置边界

## 4. 本阶段明确不做

- 不做登录 / 注册
- 不做数据库
- 不做多用户
- 不做商业化页面
- 不做支付 / 订阅
- 不做配置中心
- 不做 provider key 编辑
- 不做项目 domain 重构
- 不做 review 系统重构
- 不做第二轮大规模 UI 美化
- 不做新的大功能

## 5. 当前重点整理问题

- 文案、按钮、状态名、下载名在不同页面中还不够统一
- 当前任务、项目详情、我的项目、review 页之间的信息优先级还不完全一致
- 结果区、下载区、日志区的关系还不够像稳定产品，而更像技术页面拼接
- fallback 与 internal 入口仍需要更清楚地降级，不应与主入口抢主视野
- 设置页当前边界是对的，但说明还可以更清晰

## 6. 验收标准

- 用户在核心页面中看到的标题、状态、审核、结果、下载、按钮文案保持一致
- 核心页面的信息优先级一致，用户更容易理解“当前状态”和“下一步做什么”
- 项目详情中的结果、下载、日志关系更清楚，下载项命名更贴近用户结果
- 主入口、fallback、internal 三层入口职责清楚，默认视线聚焦主入口
- 设置页继续保持说明页定位，并能清楚表达当前支持 / 不支持与系统级配置边界
