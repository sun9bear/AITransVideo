# Trae 产出 -> 项目开发者审核 -> Claude Code 实现 三段式工作流模板

> 适用场景：当前仓库进入“商业化 + 前端 + 认证 + 支付”改造期，需要同时吸收多模型的长处，但最终仍以当前代码现实和阶段性稳定交付为最高优先级。

## 1. 目标

这套模板的目标不是“让多个 AI 同时改代码”，而是把不同模型放在最适合的位置：

- `Trae` 负责前端表达、页面结构、产品文案、Stitch 协同、方案草稿。
- `项目开发者` 负责业务与风险拍板，决定哪些建议进入主线。
- `Claude Code` 负责最终落库代码、测试、迁移、接口与阶段性交付。
- `CodeX` 负责校验边界、验证结果和下一阶段指令。

一句话：

**Trae 负责“想清楚怎么更像一个好产品”，Claude Code 负责“把它安全地变成仓库里的真实代码”。**

### 1.1 角色速查表

- `Trae`：负责页面结构、产品文案、Stitch prompt、营销与前端表达层建议。
- `项目开发者`：负责价格、Trial、风控、支付顺序与范围取舍的最终拍板。
- `Claude Code`：负责进入仓库的最终代码实现、测试回归、迁移与阶段汇报。
- `CodeX`：负责审核边界、核验结果、拦截跑偏并给出下一阶段指令。
- `Trae -> 项目开发者 -> Claude Code -> CodeX` 是默认协作顺序，不建议跳步。
- 任何涉及真相源、认证、支付、migration、测试基线的改动，必须先经过 `项目开发者` 与 `CodeX` 双重确认。

---

## 2. 总原则

### 2.1 固定分工

- `Trae`：方案、文案、页面结构、视觉方向、Stitch prompt、前端评审。
- `项目开发者`：价格、Trial、风控、支付顺序、范围与优先级拍板。
- `Claude Code`：按计划施工、改代码、跑测试、写阶段汇报。
- `CodeX`：判断是否放行、是否打回、下一步该做什么。

### 2.2 硬规则

- 任何进入仓库的代码，最终都由 `Claude Code` 编写或收口。
- `Trae` 不直接获得主线施工权。
- 任何涉及以下内容的改动，都必须先过人工审核：
  - 真相源
  - 认证
  - 支付
  - Alembic migration
  - 测试基线
  - 计费与 Trial 规则
- 所有执行都要按阶段推进：**一阶段一审核，一阶段一放行**。

### 2.3 当前仓库的默认执行准则

- `gateway` 是商业规则入口与最终真相源承载方。
- `frontend-next` 负责消费真相源，不自定义最终商业事实。
- `main.py` 和 `pytest` 必须始终可运行。
- 测试不引入真实外部 API。
- 若建议与当前代码现实冲突，以当前代码和已批准计划为准，而不是以“更理想的通用方案”为准。

## 3. 三段式工作流

## 阶段 A：Trae 产出

### A.1 适合交给 Trae 的任务

- 首页、定价页、Trial 页的信息架构。
- Hero、CTA、FAQ、价值主张文案。
- Stitch prompt 与页面分区建议。
- 营销页、注册页、升级页、Billing 页的信息层级建议。
- 视觉风格、模块顺序、移动端折叠策略。
- 轻量 API contract 草稿。

### A.2 不适合交给 Trae 的任务

- `gateway/billing.py`
- `gateway/job_intercept.py`
- `gateway/models.py`
- 认证主线
- 订阅迁移
- 支付模型迁移
- Alembic migration
- 测试主线回归

### A.3 给 Trae 的标准提示词模板

```text
请只做方案产出，不要修改代码。

你必须遵守：
1. 只基于当前仓库事实回答。
2. 不要新造价格、套餐、API 路径、数据模型。
3. 如果无法从当前代码确认，明确写“不确定”。
4. 只输出：
   - 当前现状
   - 2-3 个方案
   - 推荐方案
   - 风险与边界
   - 可交给 Claude Code 的实现说明
5. 不要直接输出最终落库代码，除非明确要求。
```

### A.4 Trae 产出的合格标准

- 引用的文件必须真实存在。
- 不擅自修改已批准计划的边界。
- 不把长期蓝图写成当前立即执行项。
- 不擅自发明新价格、新套餐权益、新 API 命名。
- 方案中必须明确“不确定点”和“风险点”。

---

## 阶段 B：项目开发者审核

这一阶段的目标不是重写方案，而是决定“能不能进入主线”。

### B.1 项目开发者重点审核什么

#### 业务与商业

- 价格是否合理。
- Trial 是否会被薅。
- Free / Plus / Pro 是否表达清楚。
- 支付顺序是否现实。

#### 范围控制

- 有没有把小任务写成大改造。
- 有没有偷带 Team / Enterprise / usage ledger。
- 有没有把后置任务提前。

#### 代码贴合度

- 是否引用了真实文件。
- 是否和当前仓库现状冲突。
- 是否擅自引入新路径、新模型、新真相源。

### B.2 直接打回的典型情况

出现以下任一情况，就不应直接进入 Claude Code 实现：

- 擅自改变 API 路径风格。
- 擅自改变价格、套餐、配额。
- 把展示逻辑写成业务真相源。
- 把 Task 0 说成 Task 3/4/5。
- 引用了仓库中不存在的函数、文件、字段。
- 明显使用“常见框架写法”代替“当前代码事实”。

### B.3 项目开发者审核后的固定结论模板

- `通过，可进入 Claude Code 实现`
- `部分通过，先修以下问题再进实现`
- `不通过，原因是边界/证据/范围不合格`

建议在结论里同时写清：

- 哪些建议被采纳
- 哪些建议被丢弃
- 本轮只允许进入哪个 Task

---

## 阶段 C：Claude Code 实现

这是唯一允许进入仓库主线代码的阶段。

### C.1 适合交给 Claude Code 的任务

- 真相源统一
- `GET /api/plans`
- 路由分组与布局拆分
- API contract 落地
- 认证改造
- Trial 发放逻辑
- 支付与订阅
- Alembic migration
- 测试编写与回归
- 文档收口

### C.2 给 Claude Code 的标准执行提示词模板

```text
请按“分阶段执行、阶段完成后停下等待审核”的方式工作。

你要执行的计划文件是：
<计划文件绝对路径>

这次只允许执行：
<Task 名称>

严格要求：
1. 只聚焦本 Task。
2. 不要提前进入后续 Task。
3. 如果计划与当前代码现状不符，先停止并汇报。
4. 不要顺手修改无关代码。
5. 若适用，先写失败测试，再做最小实现。
6. 完成后必须输出阶段完成汇报，并停止等待审核。

本轮允许修改的文件：
- ...
- ...

不要修改：
- ...
- ...

建议验证命令：
- ...
- ...
```

### C.3 Claude Code 的阶段汇报最少应包含

- 执行范围
- 读取与判断
- 实际修改
- 新增/修改的测试
- 实际执行的命令
- 验证结果
- 风险与权衡
- 产出文件
- 验收结论
- 是否已停止

### C.4 Claude Code 的合格标准

- 只修改授权文件。
- 实际运行了验证命令。
- 汇报结论基于真实结果，不夸大。
- 不把“定义 catalog”说成“订阅系统完成”。
- 不把“静态页面”说成“前后端已打通”。

---

## 4. CodeX 的职责

每轮 Claude Code 交付后，`CodeX` 负责三件事：

### 4.1 查边界

- 有没有超范围。
- 有没有把后续任务偷带进来。
- 有没有改错文件。

### 4.2 查证据

- 测试是否真的跑了。
- 基线是否保持。
- 汇报与代码事实是否一致。

### 4.3 决定下一步

- 放行
- 打回修订
- 给下一 Task 指令

## 5. 任务分流清单

### 5.1 只给 Trae

- 首页信息架构
- 定价页文案
- Trial 页面文案
- FAQ
- Stitch prompt
- Hero / CTA / Section 设计
- 前端视觉评审

### 5.2 Trae 可辅助，但不直接落库

- marketing 页组件结构建议
- pricing 卡片层级建议
- onboarding 流程建议
- Billing 页面信息结构建议

### 5.3 必须给 Claude Code

- `gateway` 真相源统一
- `GET /api/plans`
- `billing.py` / `job_intercept.py` 收口
- `frontend-next` 布局拆分
- 认证
- Trial 发放
- 支付
- subscription
- migration
- 测试

### 5.4 必须由项目开发者拍板

- 最终价格
- Trial 时长与额度
- Free 是否保留
- 支付渠道顺序
- 微信是否后置
- 风控强度
- Team / Enterprise 何时进入主线

---

## 6. 一次完整循环的标准步骤

### Step 1：Trae 先出建议

典型任务：

- “给我首页与定价页结构方案”
- “给我 Trial 页面文案”
- “给我一版 Stitch prompt”
- “给我 Billing 页面怎么讲更像 SaaS”

### Step 2：项目开发者先筛一轮

判断：

- 哪些建议有价值
- 哪些只是通用空话
- 哪些和当前仓库边界冲突

### Step 3：CodeX 写执行指令

把已批准方案收缩成：

- 只做哪个 Task
- 只改哪些文件
- 跑哪些测试
- 完成后怎么汇报

### Step 4：Claude Code 实现

它负责：

- 改代码
- 跑测试
- 输出阶段汇报

### Step 5：CodeX 复核

最后决定：

- 放行
- 打回
- 给下一步

---

## 7. 各方最适合的定位

### Trae

更适合：

- 做“像一个好产品”的部分
- 产出页面结构、文案、视觉方向
- 做前端和营销层辅助评审

不适合：

- 主线代码施工
- 真相源统一
- 支付、认证、迁移

### Claude Code

更适合：

- 做“仓库里真的能稳运行”的部分
- 严格按计划施工
- 收口为可提交代码

### 项目开发者

更适合：

- 拍板价格、Trial、风控、支付策略
- 决定当前周期该不该做、做到哪一步

### CodeX

更适合：

- 控节奏
- 审核边界
- 拦截跑偏
- 输出下一阶段指令

## 8. 最简可复制模板

### 8.1 发给 Trae

```text
请只做方案建议，不要修改代码。
只基于当前仓库事实回答。
不要新造价格、套餐、API 路径、数据模型。
输出：
1. 当前现状
2. 2-3 个方案
3. 推荐方案
4. 风险与边界
5. 可交给 Claude Code 的实现说明
```

### 8.2 发给 CodeX

```text
这是 Trae 的建议，请 CodeX 结合当前仓库现状审核：
1. 哪些可吸收
2. 哪些需要修正
3. 是否可以进入实现
4. 如果可以，请给 Claude Code 下一阶段指令
```

### 8.3 发给 Claude Code

```text
请按“分阶段执行、阶段完成后停下等待审核”的方式工作。
这次只允许执行：<Task>
只改这些文件：...
不要改这些文件：...
跑这些验证：...
完成后按固定格式输出阶段完成汇报，并停止。
```

---

## 9. 推荐使用口径

当前仓库最稳的协作方式不是“谁都能写代码”，而是：

1. `Trae` 先产出页面/文案/结构建议
2. `项目开发者` 和 `CodeX` 先筛掉跑偏部分
3. `Claude Code` 再把批准后的内容变成代码
4. 每个阶段都要跑验证并做独立审核

换句话说：

**用 Trae 吸收灵感，用 Claude Code 交付代码，用 CodeX 控制节奏。**
