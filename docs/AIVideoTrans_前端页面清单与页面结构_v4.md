# AIVideoTrans 前端页面清单与页面结构 v4

## 一、产品层分区

前端先分成两层，不混在一起：

### 1. 用户端主平台
给最终用户使用，围绕视频翻译任务流。

### 2. 内部管理/诊断入口
给你自己或内部人员看运行状态、排错、日志、配置状态。  
这个先做成**次级入口**，不要做成主导航中心。

---

## 二、一级导航

用户端主导航建议固定为：

- 新建翻译
- 当前任务
- 审校处理
- 我的项目
- 设置

内部入口建议先不进主导航，放到：
- 设置页底部
- 或单独 `/internal` / `/admin` 路径

---

# 三、页面清单

---

## A. 新建翻译

### 页面定位
用户进入平台后的默认主入口。

### 页面目标
最快速发起一个新任务，同时让用户清楚当前限制。

### 页面模块

#### 1. 页面头部
- 标题：新建翻译
- 简短说明：输入视频链接，开始生成翻译项目

#### 2. 输入卡片
- YouTube URL 输入框
- 提交按钮
- URL 合法性即时提示
- 当前是否支持该输入类型的说明

#### 3. 基础参数卡片
只保留当前真实支持且有价值的参数：
- 说话人数
  - 自动
  - 1 人
  - 2 人
- Voice A
- Voice B

#### 4. 当前运行中任务提示
如果当前已有 active job：
- 提示已有任务运行中
- 跳转按钮：去当前任务
- 禁止重复提交或弱化提交入口

#### 5. 当前限制说明
展示当前基线边界：
- 当前只支持 YouTube URL
- 当前单任务运行
- 处理中可能进入审校
- 完成后到“我的项目”查看结果

### 页面状态
- 空状态
- 输入中
- 校验失败
- 提交中
- 提交成功
- 已有任务运行中

---

## B. 当前任务

### 页面定位
当前单个活跃任务的控制台。

### 页面目标
让用户一眼知道：
- 现在正在跑什么
- 当前配置是什么
- 卡在哪一步
- 是否需要我处理
- 下一步去哪

### 页面模块

#### 1. 顶部状态条
- 当前任务标题 / 视频标题 / URL 摘要
- 状态 badge
  - queued
  - running
  - waiting for review
  - succeeded
  - failed
- 当前阶段
- 创建时间
- 最近更新时间

#### 2. 当前配置卡片
- source type
- URL
- speakers
- voice_a
- voice_b
- output target
- 配置来源说明（默认/用户选择）

#### 3. 进度阶段卡片
用用户可理解的阶段表达：
- 输入准备
- 媒体理解
- 说话人审校
- 翻译审校
- 草稿与配音
- 输出完成

不要直接暴露内部 S0/S1。

#### 4. 日志卡片
- 默认显示最近关键日志
- 展开查看完整日志
- 失败时自动高亮错误片段

#### 5. 审校提示卡片
仅在 waiting_for_review 时显示：
- 当前需要处理什么 review
- 一句说明为什么要处理
- 主按钮：去处理

#### 6. 结果摘要卡片
仅在 succeeded 时显示：
- manifest 是否可用
- 主要输出项数量
- 跳转按钮：查看项目结果

#### 7. 失败摘要卡片
仅在 failed 时显示：
- 失败阶段
- 简短失败原因
- 查看日志
- 去我的项目详情

### 页面状态
- 无当前任务
- queued/running
- waiting_for_review
- succeeded
- failed

---

## C. 审校处理

### 页面定位
当前所有待处理 review 的集中入口。

### 页面目标
把所有“必须用户介入才能继续”的任务集中展示。

### 页面结构
建议拆成两个层级：

### C1. 审校列表页
展示：
- 项目名
- review 类型
- 等待时间
- 当前状态
- 进入处理按钮

review 类型：
- speaker review
- translation review
- voice review

---

### C2. 审校详情页
按 review 类型渲染不同内容。

---

### C2-1. Speaker Review 页面

#### 模块
- 项目基础信息
- 当前识别出的说话人列表
- 每个说话人的命名/归属
- 必要时展示样本片段或摘要
- approve / continue 按钮

#### 目标
让用户确认“谁是谁”。

---

### C2-2. Translation Review 页面

#### 模块
- 原文片段
- 当前译文
- 可编辑或确认区域
- 说明：确认后继续流程
- approve / continue

#### 目标
让用户确认翻译是否可接受。

---

### C2-3. Voice Review 页面

#### 模块
- 当前音色方案
- 预览说明
- approve / continue
- 如当前系统支持，提供最小重试/重新选择入口
- 如不支持，不伪装支持

#### 目标
让用户确认音色是否可接受。

---

## D. 我的项目

### 页面定位
历史项目中心。

### 页面目标
查看所有历史项目，进入详情，回看结果。

### 页面模块

#### 1. 顶部筛选区
- 搜索框
- 状态筛选
  - 全部
  - 运行中
  - 待审校
  - 已完成
  - 失败

#### 2. 项目列表
每个项目卡片/行展示：
- 项目标题
- 来源摘要
- 状态
- 最近更新时间
- 是否待审校
- 是否已有结果

#### 3. 快捷操作
- 查看详情
- 继续处理
- 查看结果
- 下载结果（如已完成）

### 页面状态
- 空项目状态
- 有项目但无完成项
- 已完成项目列表
- 失败项目列表

---

## E. 项目详情

### 页面定位
单个项目的主信息页。  
这里是“结果与下载”的最终落点。

### 推荐结构
用 Tabs 更清楚。

#### Tab 1：概览
- 项目标题
- 来源 URL
- 状态
- 当前阶段
- 创建时间
- 最近更新时间
- 当前摘要

#### Tab 2：进度与日志
- 阶段流转
- 最近日志
- 完整日志入口
- 错误摘要
- fallback 摘要（如存在）

#### Tab 3：审校记录
- speaker_review 状态
- translation_review 状态
- voice_review 状态
- 当前是否还待处理
- 已批准记录

#### Tab 4：结果与下载
- manifest 是否可用
- output 数量
- 白名单下载区
  - manifest.file
  - translation.segments
  - editor.subtitles
  - editor.dubbed_audio_complete
  - publish.dubbed_video
- 每个产物的一句说明

### 页面目标
用户围绕某个项目完成：
- 回看
- 判断状态
- 进入审校
- 获取结果

---

## F. 设置

### 页面定位
最小个人设置与说明页。

### 页面模块
- 默认 speakers 设置
- 默认 voice 偏好
- 当前版本说明
- 当前稳定基线说明
- 当前限制说明
- 内部入口链接（弱入口）

### 暂不放入
- provider key 明文编辑
- 大量服务端配置
- 部署参数
- 复杂权限系统

---

# 四、内部管理/诊断入口

这个不放主导航，建议作为独立次级入口。

## G. 系统状态页
展示：
- Web UI 状态
- Job API 状态
- public entry 状态
- 当前是否有 active job

## H. 日志诊断页
展示：
- runtime logs
- access logs
- 最近错误摘要

## I. 配置状态页
展示：
- 是否已配置必要环境变量
- 当前 provider 状态
- 仅展示是否配置，不展示密钥值

---

- `/current`
- `/reviews`
- `/reviews/:projectId/:reviewType`
- `/projects`
- `/projects/:projectId`
- `/settings`
- `/internal/status`

# 七、组件层建议

为了后面开发顺一点，页面里建议沉淀这些共用组件：

- `StatusBadge`
- `StageProgress`
- `ProjectCard`
- `ReviewTaskCard`
- `LogViewer`
- `ResultDownloadList`
- `EmptyState`
- `ConfigSummaryCard`

---

# 八、哪些内容现在不要做

这轮页面细化里，先不要加：

- 用户账户系统
- 套餐/支付
- 多项目并发控制 UI
- 高级分析图表
- 完整文件浏览器
- 任意路径下载
- failed job resume UI
- Skill 专属入口
- 团队协作页面

---

# 九、最重要的产品原则

这套前端不是“后台管理系统”，而是：

**一个以任务推进和项目回看为核心的视频翻译工作台。**

所以页面排序要始终服从这条主线：

- 用户先发起
- 再盯当前任务
- 需要时做审校
- 最后在项目里看结果和下载
