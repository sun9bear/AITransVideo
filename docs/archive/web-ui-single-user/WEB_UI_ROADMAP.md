# Web UI 后续功能路线图

建议使用 UTF-8 编码打开本文件。

## 目标

Web UI 不再只作为“启动任务”的页面，而是逐步演进成一个按流程分阶段操作的本地审校工作台。

核心方向：

- 用标签页拆分不同流程和功能区，避免单页堆叠失控
- 用分页和筛选承接大量段落数据
- 逐步补齐“查看 -> 确认 -> 修改 -> 局部重跑 -> 再确认”的闭环

## 总体设计原则

- 先做可见性，再做可编辑性
- 先做结果集中展示，再做逐段审校
- 先做单条能力，再考虑批量能力
- 先做局部重跑，再考虑更复杂的流程自动化
- Web UI 继续复用现有 CLI/service/workflow 边界，不单独长出第二套后端

## 标签页规划

建议最终按以下标签页组织：

- `运行`
- `设置`
- `转录与发言人`
- `翻译与重写`
- `音色与语音库`
- `音频试听与对齐`
- `结果`

当前最小可落地版本建议先做：

- `运行`
- `设置`
- `结果`
- `审校占位`

## 分页与列表规则

段落较多时，必须支持分页显示。

建议规则：

- 默认每页显示 `20` 条
- 可切换 `20 / 50 / 100`
- 支持按段号搜索
- 支持跳转到指定页或指定段
- 顶部显示总段数、当前页、待处理数量
- 所有段落类页面复用同一套分页和筛选能力

建议通用筛选维度：

- `speaker`
- `needs_review`
- `已确认 / 未确认`
- `已重写 / 未重写`
- `已试听 / 未试听`
- `对齐正常 / 对齐异常`

## 近期

目标：先把“看得见、找得到、定位快”做好。

- 搭建标签页骨架，先拆分为 `运行`、`设置`、`结果`、`审校占位`
- `运行` 页保留现有任务启动、日志、运行状态和阶段信息
- `设置` 页继续承载 provider、prompt、模型路由等配置
- `结果` 页明确展示 `manifest`、editor 产物、publish 产物和关键文件入口
- 增加 `needs_review` 集中展示区块，先做只读列表
- 增加通用分页能力，默认每页 `20` 条
- 增加通用筛选和段号搜索能力
- 为后续段落页统一列表行结构

建议每条段落先显示这些字段：

- 段号
- 发言人
- 当前状态
- 文本摘要
- 是否 `needs_review`
- 音频/字幕/对齐入口摘要

## 中期

目标：进入“逐段确认”和“语音资产管理”。

### 转录与发言人

- 查看转录稿
- 修改发言人名字
- 逐段确认发言人
- 支持批量确认

### 翻译与重写

- 查看 literal translation
- 查看 TTS rewrite 文本
- 逐段确认翻译稿
- 单条选择是否重写
- 对不合格段落执行批量自动重写

### 音色与语音库

- 每个 speaker 选择音色
- 音色库增删改
- 人名修改与 speaker 绑定调整
- 默认音色设置

### 音频试听与对齐

- 逐段试听
- 单条重新 TTS
- 显示目标时长、实际时长、偏差
- 集中处理 `needs_review`

## 后期

目标：形成“局部修正 + 版本比较 + 结果复核”的完整闭环。

- 支持单条局部重跑
- 支持单条 `translation / rewrite / TTS / alignment` 重跑
- 支持批量重写和批量重 TTS
- 支持版本对比
- 支持运行历史回看
- 支持审校状态持久化
- 支持更完整的结果页和 manifest 审计视图
- 需要时再考虑波形、时间轴可视化和更多批量操作

建议审校状态至少包含：

- `待确认`
- `已确认`
- `已跳过`
- `需返工`

## 当前最值得优先推进的顺序

1. 做标签页骨架
2. 做结果页和 `needs_review` 集中展示
3. 做通用分页和筛选
4. 做“转录与发言人”确认页
5. 做“翻译与重写”确认页
6. 做“音色与语音库”页
7. 做“音频试听与对齐”页
8. 最后再做局部重跑和批量修正

## 当前不建议优先做

- 复杂时间轴编辑器
- 大而全单页 UI
- 多用户后台
- 任务队列系统
- 过早把 Web UI 做成“完整剪映替代品”

## 备注

这份路线图面向当前项目流程：

- `media_understanding`
- `translation`
- `chunking`
- `alignment`
- `editor output`
- `publish output`

后续 Web UI 的增强，应始终围绕现有 canonical build 和统一 output layer 展开，而不是重新长出一条独立流程。
