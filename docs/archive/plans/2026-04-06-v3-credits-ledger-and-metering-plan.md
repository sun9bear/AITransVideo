# V3 点数体系、用量账本与精确计量方案

> 状态：方案草案  
> 时间：2026-04-06  
> 适用阶段：V2 主线基本完成后的下一阶段商业化升级  
> 明确排除：**WeChat Pay 不在本方案范围内**

---

## 1. 目标

V3 的目标不是继续扩前端页面，而是把当前较为分散的商业化口径：

- Free 次数
- Trial 分钟
- 套餐分钟
- 手工补偿

统一升级为一套可计费、可审计、可退款、可扩展的 **点数（Credits）+ 用量账本（Usage Ledger）** 体系。

核心结果：

1. 用 **点数** 作为统一消耗单位。
2. 用 **账本** 记录每一笔发放、预扣、结算、返还、补偿。
3. 用 **分钟级精确计量** 作为任务完成后的真实结算基础。
4. 用 **退款后的 entitlement rollback** 保证支付状态、点数状态、可用权限三者一致。

---

## 2. 设计原则

### 2.1 Gateway 继续是真相源

V3 仍然遵守现有架构规则：

- Gateway 是套餐、Trial、点数规则、权限规则的真相源
- Frontend 只消费余额、预计消耗、账单和状态
- 前端不得自己计算最终扣点真相

### 2.2 点数体系完全替代旧口径

V3 明确采用以下方向：

- **点数体系完全替代现在的 Free 次数 / Trial 分钟口径**
- 用户对外只看到：
  - 剩余点数
  - 本次预计消耗点数
  - 本次实际消耗点数

不再长期并行展示：

- Free 次数
- Trial 分钟
- 套餐分钟

### 2.3 对外表达简化，对内成本模型保留

对外：

- 使用 `点数 / Credits`
- 使用 `快捷版 / 工作台版`
- 使用 `标准 / 高质 / 旗舰` 这样的档位表达

对内：

- 仍保留真实模型、真实供应商、真实成本来源
- 扣点规则通过 Gateway 配置映射

### 2.4 先记账，再谈高级计费

V3 的第一优先级不是复杂优惠，而是：

- 账记对
- 扣得明白
- 退款能回滚
- 管理员能审计

---

## 3. 统一额度来源模型

V3 将额度来源统一为 4 类：

### 3.1 Free Credits

- 面向免费用户
- 作为基础体验额度
- 主要服务 `快捷版`

### 3.2 Trial Credits

- 面向新用户试用
- 与 Free 分开建账
- 带有效期
- 主要服务 `工作台版`

### 3.3 Subscription Credits

- 随 Plus / Pro 周期发放
- 作为订阅权益的一部分

### 3.4 Top-up Credits

- 单独购买的充值点数
- **免费用户和付费订阅会员都可以单独购买**
- 默认建议不过期

---

## 4. 当前已确认的点数方向

以下为当前已确认的 V3 方向性规则，后续仍需冻结成正式商业真相：

### 4.1 Trial Credits

- `Trial = 300 credits`
- `7 天有效`
- 单独建账，不混入其他余额

### 4.2 Free Credits

- `Free = 500 credits`

### 4.3 消耗优先顺序

#### 快捷版（Express）

- 优先消耗 `Free credits`
- Free 不足时，再消耗 `Top-up / Subscription credits`
- 不优先动 `Trial credits`

#### 工作台版（Studio）

- 优先消耗 `Trial credits`
- Trial 用完或过期后，再消耗 `Top-up / Subscription credits`
- 不优先动 `Free credits`

---

## 5. 扣点模型

V3 的核心不再是“给多少分钟”，而是“每分钟按什么规则扣多少点”。

### 5.1 扣点维度

建议至少按以下维度计算：

- 使用模式：`快捷版 / 工作台版`
- 质量档位：`标准 / 高质 / 旗舰`
- 实际源视频分钟数
- 是否发生重跑 / 增量重生成

### 5.2 推荐规则结构

建议采用：

`扣点 = 实际计费分钟 × 模式倍率 × 质量倍率`

这样有几个好处：

- 模型切换时不需要改前端文案
- 供应商切换时不暴露给用户
- 将来增加新模型、新质量档位时不必重写账本结构

### 5.3 预计扣点与实际扣点分离

任务执行前：

- 计算 `estimated_credits`

任务完成后：

- 按真实分钟数 / 真实路径结算 `actual_credits`

两者差额：

- 可补扣
- 可返还

## 5.4 每 1 分钟源视频的人民币成本公式

V3 点数定价的基础，不应直接从营销口径反推，而应先建立一套统一的 **每 1 分钟源视频人民币成本公式**。

当前阶段先采用以下边界：

- 只计算 **1 个目标语种**
- 当前只考虑 **中文输出**
- 不考虑音色克隆
- 翻译大模型、S2 审校、重写额外支出、服务器成本均按 **每分钟固定摊销**
- TTS 成本按不同模型单独变化

说明：

- 当前 `翻译 / S2 审校 / 重写 / 服务器` 的固定摊销属于 **安全冗余估算**
- 其作用是给第一版点数定价留出保守安全边界，而不是宣称这些就是当前真实结算成本
- V3 上线后应以真实 ledger 数据在 `2-4 周` 内做一次回测校准

### 5.4.1 核心公式

设：

- `C_asr` = 每 1 分钟转录成本
- `C_translate` = 每 1 分钟翻译固定摊销
- `C_s2_review` = 每 1 分钟 S2 审校固定摊销
- `C_rewrite` = 每 1 分钟重写固定摊销
- `C_server` = 每 1 分钟服务器固定摊销
- `C_tts(model)` = 指定 TTS 模型的每 1 分钟 TTS 成本

则：

```text
每 1 分钟源视频总成本(model)
= C_asr + C_translate + C_s2_review + C_rewrite + C_server + C_tts(model)
```

### 5.4.2 关键参数定义

#### `K_cn_chars_per_src_min`

含义：

- 每 1 分钟源视频
- 最终产出的中文配音文本
- 平均大约对应多少个中文字符

它本质上是一个：

- `源视频分钟 -> 中文字符数`

的换算系数。

当前推荐估算范围：

- 低语速 / 信息密度低：`180 - 220`
- 常见中文讲解 / 口播：`220 - 280`
- 快语速 / 信息密度高：`280 - 340`

当前推荐中值：

```text
K_cn_chars_per_src_min = 250
```

理由：

- 不像 `200` 那么偏保守
- 也不像 `300+` 那么偏激进
- 适合作为第一版成本和点数测算的中位值

#### `FX_usd_cny`

含义：

- 美元兑人民币汇率
- 用于把美元计费服务换算成人民币成本

当前第一版估算使用：

```text
FX_usd_cny = 7
```

### 5.4.3 各成本项拆解

#### ASR 转录成本

当前建议按 `AssemblyAI Universal-2 + Speaker Diarization` 估算：

```text
C_asr = ((P_asr_usd_per_hour + P_diarization_usd_per_hour) × FX_usd_cny) / 60
```

当前参考价：

- `P_asr_usd_per_hour = 0.15`
- `P_diarization_usd_per_hour = 0.02`

则：

```text
C_asr = ((0.15 + 0.02) × 7) / 60
      = 0.0198 元/分钟
```

#### 翻译 / S2 / 重写 / 服务器固定摊销

当前采用第一版运营估算占位值（**安全冗余估算**）：

```text
C_translate = 0.03 元/分钟
C_s2_review = 0.02 元/分钟
C_rewrite   = 0.02 元/分钟
C_server    = 0.03 元/分钟
```

选择固定摊销的原因：

- 当前阶段还没有完整 usage ledger 支持概率型精细核算
- 先把定价口径稳定下来，比过度精细更重要
- 后续可在 ledger 成熟后改成更精确的真实结算/概率模型

#### TTS 成本

##### 豆包

当前先按豆包接入商品线的实际单价做字符成本换算：

```text
C_tts(doubao) = K_cn_chars_per_src_min / 10000 × P_doubao_rmb_per_10k_chars
```

##### MiniMax

MiniMax 官方按字符计费，当前口径中：

- `1 个汉字 = 2 个计费字符`

因此：

```text
C_tts(minimax) = (2 × K_cn_chars_per_src_min) / 10000 × P_minimax_rmb_per_10k_chars
```

##### CosyVoice

```text
C_tts(cosyvoice) = (2 × K_cn_chars_per_src_min) / 10000 × P_model_rmb_per_10k_chars
```

阿里云百炼当前中文同样按 **2 字符** 计费。

## 6. Usage Ledger（用量账本）

### 6.1 为什么需要账本

当前系统即便有 Trial / Subscription / Billing，也还缺一层真正的“用量流水”。

V3 需要一套独立账本，用来回答：

- 哪个用户什么时候获得了多少点数
- 哪次任务预扣了多少
- 哪次任务实际结算了多少
- 哪次失败返还了多少
- 哪次退款回退了多少权限和额度
- 哪次管理员补偿了多少

### 6.2 建议最小账本记录

每一条 ledger entry 至少记录：

- `user_id`
- `bucket_type`
  - `free`
  - `trial`
  - `subscription`
  - `topup`
  - `manual_adjustment`
- `direction`
  - `grant`
  - `reserve`
  - `capture`
  - `release`
  - `refund`
  - `rollback`
- `credits_delta`
- `balance_after`
- `related_job_id`
- `related_order_id`
- `related_subscription_id`
- `reason_code`
- `created_at`

### 6.3 Bucket（额度桶）概念

建议不要只维护一个总余额，而是维护多个“额度桶”：

- Free bucket
- Trial bucket
- Subscription bucket
- Top-up bucket

每个 bucket 自己带：

- `granted_credits`
- `remaining_credits`
- `expires_at`
- `source`

这样才能自然支持：

- Trial 7 天过期
- 充值点数不过期
- 订阅点数按周期刷新
- 不同来源不同消耗优先级

## 7. 分钟级精确计量

### 7.1 为什么 V3 要引入分钟级精确计量

当前很多商业口径还是“分钟”概念，但系统实际并没有完整、稳定地把每次任务的真实可计费分钟沉淀为统一账本事实。

V3 需要把：

- 预估分钟
- 实际分钟
- 增量重生成分钟

统一到同一套任务结算模型里。

### 7.2 推荐结算流程

1. 用户创建任务
2. 系统估算源视频分钟并预估点数
3. 在 ledger 中做一笔 `reserve`
4. 任务完成后，生成真实 `actual_billable_minutes`
5. 按真实分钟和质量档位做 `capture`
6. 未使用部分 `release`
7. 失败任务按规则全返或部分返还

### 7.3 计量边界建议

V3 只建议先做：

- `source video minutes`
- `实际处理分钟`
- `增量重生成分钟`

不建议一开始就混入：

- 每个子模型的 token 级成本
- 每个小步骤的细碎内部成本

先把用户能理解、账本能闭环的分钟级事实做稳。

## 8. 退款后的 Entitlement Rollback

### 8.1 问题定义

V2 目前已经有 payment / invoice / subscription 的基础模型，但退款后的完整权限回滚还没有产品化。

V3 需要解决：

- 用户退款后，是否还保留权限
- 已发放的订阅点数怎么办
- 已消耗的点数怎么办
- 部分退款和整单退款如何处理

### 8.2 V3 建议目标

退款不是只改订单状态，而是同步影响 3 个层次：

1. `billing truth`
2. `credits ledger`
3. `effective entitlements`

### 8.3 推荐回滚规则

#### 情况 A：退款时点数尚未消耗

- 直接回滚剩余额度
- 更新 subscription / entitlements

#### 情况 B：退款时部分点数已消耗

- 不直接允许产生负无穷余额
- 需要进入明确策略之一：
  - 标记账户为 `credits_overdrawn`
  - 降级但允许继续查看历史
  - 下次充值/订阅优先抵扣负余额

#### 情况 C：整单退款 + 周期权限撤销

- 订阅状态转为已退款/已取消
- 相关未消耗 subscription credits 回滚
- entitlements 根据剩余有效 bucket 重新计算

### 8.4 为什么要产品化

如果没有完整 rollback，系统会出现：

- 订单显示已退款
- 但权限还在
- 或点数还在
- 或 UI 和 Gateway 状态不一致

这会直接破坏商业可信度。

## 9. Frontend 展示原则

Frontend 在 V3 只消费事实，不创造事实。

建议新增或改造的核心展示有：

- 当前总可用点数
- 各来源点数明细
  - Free
  - Trial
  - Subscription
  - Top-up
- Trial 剩余有效期
- 本次任务预计消耗点数
- 任务完成后的实际扣点记录
- 退款/回滚后的状态说明

明确禁止：

- 前端自行计算最终扣点
- 前端自行判定最终余额
- 前端把 Trial / Free / Subscription 逻辑写死为最终真相

## 9.1 点数与人民币换算建议

### 9.1.1 成本承载锚点

建议先定义：

```text
1 点 ≈ 0.015 元成本承载
```

理由：

- 刚好落在当前每分钟基础成本区间可接受的折算范围内
- 可以让 `10 点/分钟`、`15 点/分钟` 这类规则较自然地贴合基础成本
- 兼顾后续高质量档位的加价空间

按该锚点，前述每分钟基础成本大致可映射为：

| 模型/档位 | 每分钟基础成本 | 折算成本点数 |
|---|---:|---:|
| CosyVoice 海外版 | 0.2152 | 约 14 点 |
| 豆包（保守口径） | 0.2198 | 约 15 点 |
| CosyVoice 国内版 | 0.2198 | 约 15 点 |
| MiniMax 2.8 turbo（按量） | 0.2198 | 约 15 点 |
| MiniMax 2.8 hd（按量） | 0.2948 | 约 20 点 |

### 9.1.2 对外售价锚点

当前约束是：

- 通用额外购买点数先按 **50% 毛利率** 测算

则：

```text
售价 = 成本 / (1 - 0.5) = 成本 × 2
```

因此推荐第一版：

```text
1 点对外售价 ≈ 0.03 元
```

即：

- `1 点成本承载 ≈ 0.015 元`
- `1 点建议售价 ≈ 0.03 元`

### 9.1.3 与当前讨论过的扣点规则的关系

在上述锚点下：

- `10 点 / 分钟` ≈ `0.30 元 / 分钟售价`
- `15 点 / 分钟` ≈ `0.45 元 / 分钟售价`
- `30 点 / 分钟` ≈ `0.90 元 / 分钟售价`
- `50 点 / 分钟` ≈ `1.50 元 / 分钟售价`

这说明：

- `快捷版 10 点 / 分钟`
- `工作台基础版 15 点 / 分钟`

是相对合理的起始规则。

前提是：

- 快捷版必须绑定经过采购优化和成本校准的低成本 TTS 池
- 不应默认让快捷版直接落到高成本 TTS 路径

而：

- `30 点 / 分钟`
- `50 点 / 分钟`

更适合作为高质/旗舰档，而不应长期直接绑定底层模型名。

## 9.2 Top-up 点数包定价建议

### 9.2.1 定价目标

Top-up 点数包的目标是：

- 免费用户和订阅用户都能单独购买
- 不打穿订阅价值
- 保持 50% 毛利率底线
- 更贴合中文 SaaS 用户的常见价格心智

### 9.2.2 当前推荐方案（推荐采用）

#### 方案 A：稳妥版

| 点数包 | 建议售价 |
|---|---:|
| 1000 点 | ¥39 |
| 3000 点 | ¥99 |
| 5000 点 | ¥159 |
| 10000 点 | ¥299 |

对应单点售价约为：

- `1000 点`：`0.039 元/点`
- `3000 点`：`0.033 元/点`
- `5000 点`：`0.0318 元/点`
- `10000 点`：`0.0299 元/点`

推荐理由：

- 小包不至于过低，避免免费用户长期只买最小包
- 大包逐步贴近 `0.03 元/点` 锚点
- `¥99 / ¥159 / ¥299` 更符合中文付费习惯

- 但利润余量更小

### 9.2.4 当前推荐结论

## 9.3 Free / Trial / Subscription 附赠点数建议

### 9.3.1 当前建议口径

| 来源 | 当前建议 |
|---|---:|
| Free | 500 点 |
| Trial | 300 点 / 7 天有效 |
| Plus | 3500 点 / 月 |
| Pro | 12000 点 / 月 |

### 9.3.2 理由

#### Free = 500 点

- 作为基础体验额度
- 主要服务快捷版
- 量级足够让用户理解产品，但不足以长期白嫖工作台深度能力

#### Trial = 300 点 / 7 天有效

- 用于工作台版试用体验
- 应单独建账、带过期时间
- 不与 Free 点数混用

#### Plus = 3500 点 / 月

理由：

- 对 `¥99/月` 而言，赠送 3500 点仍能明显体现订阅价值
- 相比 4000 点，能保留更健康的毛利缓冲和后续校准空间

大致可支撑：

- 快捷版：约 `350 分钟`
- 工作台基础版：约 `233 分钟`
- 高质档：约 `116 分钟`
- 旗舰档：约 `70 分钟`

#### Pro = 12000 点 / 月

理由：

- 对 `¥299/月` 而言区分度仍然明显
- 相比 15000 点，更有利于维持长期毛利和 Top-up 的存在意义

大致可支撑：

- 快捷版：约 `1200 分钟`
- 工作台基础版：约 `800 分钟`
- 高质档：约 `400 分钟`
- 旗舰档：约 `240 分钟`

### 9.3.3 当前推荐正式建议

V3 第一版建议：

- `Free = 500 点`
- `Trial = 300 点 / 7 天有效`
- `Plus = 3500 点 / 月`
- `Pro = 12000 点 / 月`

若后续扩展季付/年付，可先简单按月度点数乘以周期长度，再根据折扣策略微调。

### 9.3.4 当前建议的点数消耗优先级

#### 快捷版

- 优先消耗 `Free credits`
- 其次消耗 `Subscription credits`
- 再消耗 `Top-up credits`
- 不优先动 `Trial credits`

#### 工作台版

- 优先消耗 `Trial credits`
- 其次消耗 `Subscription credits`
- 再消耗 `Top-up credits`
- 不优先动 `Free credits`

说明：

- `Subscription credits` 建议优先于 `Top-up credits` 消耗
- 理由是充值点数默认不过期，订阅点数更适合优先使用

## 10. 对现有 V2 的迁移策略

V3 应采用 staged migration，而不是 big-bang 替换。

### 阶段 1：账本与点数桶并行落地

- 保留现有 V2 套餐和 Trial 流程
- 在后台新增 credits bucket + ledger
- 先让新任务同时写旧逻辑和新账本

### 阶段 2：前端开始展示点数

- Billing UI 展示点数余额和试用点数
- 工作台展示预计扣点

### 阶段 3：用点数真相替代旧口径

- Free 次数下线
- Trial 分钟展示下线
- 前端改为只展示点数

### 阶段 4：退款和回滚闭环

- billing / credits / entitlements 三者联动

## 13. 当前结论

V3 的核心不是“再加一个支付渠道”，而是：

- 把商业化从“套餐 + 模糊分钟口径”
- 升级为“点数 + 账本 + 精确结算 + 可回滚权限”

这会成为后续：

- 充值点数
- 订阅额度
- Trial 过期
- 退款回滚
- 更复杂模型成本分层

的统一底座。
